## 概述  [[⬆](#目录)]

Blender的文本渲染系统是一个高性能的缓存系统，用于在3D视口中显示各种文本信息，包括几何节点属性、测量信息、索引等。该系统主要分为两个部分：文本缓存添加和文本绘制。

## 核心组件分析  [[⬆](#目录)]

### 1. DRWTextStore

**定义位置**: `draw_manager_text.cc:71-73`

```cpp
struct DRWTextStore {
  BLI_memiter *cache_strings;
};
```

**功能**:
- 文本缓存的主容器
- 包含一个`BLI_memiter`用于高效存储大量文本字符串
- 通过`DRW_text_cache_create()`创建，`DRW_text_cache_destroy()`销毁

**关键特性**:
- 使用`BLI_memiter_create(1 << 14)`初始化，初始容量16KB
- 支持动态扩展，避免频繁内存分配
- 设计为批量存储，提高渲染效率

### 2. BLI_memiter

**功能**: Blender的高效内存迭代器
- 用于批量存储可变大小的数据结构
- 支持顺序访问和内存高效分配
- 避免了传统链表或动态数组的内存碎片问题

**在文本系统中的作用**:
- 存储`ViewCachedString`结构和字符串数据
- 支持变长字符串的高效存储
- 提供快速的遍历接口

### 3. ViewCachedString

**定义位置**: `draw_manager_text.cc:54-69`

```cpp
struct ViewCachedString {
  float vec[3];           // 3D世界坐标位置
  union {
    uchar ub[4];          // 颜色 (RGBA)
    int pack;
  } col;
  short sco[2];          // 投影后的屏幕坐标
  short xoffs, yoffs;    // 像素偏移
  short flag;            // 标志位
  int str_len;           // 字符串长度
  bool shadow;           // 是否显示阴影
  bool align_center;     // 是否居中对齐
  char str[0];           // 柔性数组成员，存储实际字符串

```

**设计亮点**:
- **柔性数组**: `char str[0]` 允许在结构体末尾存储变长字符串
- **联合体颜色**: 支持RGBA和紧凑整数两种颜色表示
- **3D到2D映射**: 存储3D位置和2D屏幕坐标

### 4. StringRef

**作用**: Blender的字符串引用类
- 避免不必要的字符串复制
- 提供子字符串操作
- 支持UTF-8字符串处理

## 实现中值得注意的地方  [[⬆](#目录)]

### 1. 内存管理策略

**柔性数组 (Flexible Array Member) 模式**:
```cpp
// 结构体末尾的变长数组
struct ViewCachedString {
  // ... 其他成员
  char str[0];  // 柔性数组 或者写成 char str[]
};
```

#### A. 它的核心概念
这是 C/C++ 的一种黑魔法（在 C99 中标准化，C++ 中作为扩展支持）。它允许结构体的**最后一个成员**是一个长度为 0 的数组。

*   **`sizeof(ViewCachedString)`** 计算大小时，**是不包含 `str[0]` 的**。它只计算前面那些成员的大小。
*   `str[0]` 只是一个**“路标”**，它指向结构体尾部紧接着的内存地址。

#### C. 为什么要用柔性数组？（对比指针法）

**方法一：普通指针（笨办法）**
```cpp
struct OldString {
    int len;
    char *ptr; // 指向别的地方
};
// 需要两次 malloc：
OldString *obj = malloc(sizeof(OldString));
obj->ptr = malloc(100); // 字符串存在别处，内存不连续
```
*   **缺点**：两次分配，两次释放，内存碎片化，缓存不友好。

**方法二：柔性数组（高手办法）**
```cpp
struct NewString {
    int len;
    char str[0]; // 就在屁股后面
};
// 只需要一次 malloc：
NewString *obj = malloc(sizeof(NewString) + 100);
// 字符串直接存入 obj->str，内存完全连续
```
*   **优点**：
    1.  **一次分配**：减少 `malloc` 开销。
    2.  **缓存命中极高**：CPU 读完结构体头信息，顺势就能读到后面的字符串，因为它们在物理内存上是紧挨着的。
    3.  **管理方便**：释放时直接 `free(obj)` 整个块，不用担心忘记释放 `ptr` 指向的内存。

**内存分配计算**:
申请了 `结构体本身大小` + `字符串长度` 这么大的一块连续内存
利用 **Memiter 内存池** 申请了一块 **连续内存**，大小刚好够放 **结构体头 + 字符串内容**，完美利用了**柔性数组**特性。
```cpp
alloc_len = (flag & DRW_TEXT_CACHE_STRING_PTR) ? sizeof(void *) : str_len + 1;
vos = static_cast<ViewCachedString *>(
    BLI_memiter_alloc(dt->cache_strings, sizeof(ViewCachedString) + alloc_len));
```

## 总结  [[⬆](#目录)]

Blender的文本渲染系统展现了以下优秀的设计原则：

1. **高效内存管理**: 使用`BLI_memiter`和柔性数组避免内存碎片
2. **批量处理**: 先缓存后渲染，减少GPU状态切换
3. **类型安全**: 使用模板元编程进行编译时类型检查
4. **性能优化**: 并行计算、状态缓存、投影预筛选
5. **扩展性**: 支持多种数据类型和坐标系
6. **健壮性**: 完善的错误处理和边界检查

这个系统为Blender提供了强大而高效的3D文本显示能力，是现代图形编程的优秀范例。
