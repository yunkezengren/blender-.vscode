| # | 规则 | 说明 |

## 🔄 RIPER 流程响应矩阵 🆕

### CC 当前阶段识别

| CC 关键词 | RIPER 阶段 | Codex 响应策略 | 输出重点 |
|-----------|-----------|----------------|----------|
| "研究"、"调研"、"分析"、"评估" | **Research** | 深度分析 + 技术选型对比 | 问题本质、技术可行性、风险评估 |
| "拆分"、"步骤"、"迭代"、"分解" | **Iterate** | 提供 3-5 个独立步骤 | 步骤独立性、依赖关系 |
| "规划"、"计划"、"任务"、"安排" | **Plan** | 包含测试策略 + 依赖关系 | 测试策略、验收标准 |
| "实现"、"编码"、"原型"、"写代码" | **Execute** | 具体实现 + 边界情况 | 完整 diff、边界处理 |
| "审查"、"review"、"检查"、"验证" | **Review** | 严格审查 + 反思清单 | 问题列表、反思问题 |

### 阶段特定输出要求

#### Research 阶段
```
✅ 必须包含:
  - 问题本质定义（一句话）
  - 技术选型对比表（至少 2 个方案）
  - 风险评估（高/中/低）
  - 技术可行性验证

❌ 避免:
  - 过早给出具体实现代码
  - 单一方案（应提供对比）
  - 缺少风险评估
```

#### Iterate 阶段
```
✅ 必须包含:
  - 3-5 个独立步骤
  - 步骤间依赖关系图
  - 每步的验收标准
  - 备选方案（如适用）

❌ 避免:
  - 步骤过多（>5）或过少（<3）
  - 步骤间强耦合
  - 缺少依赖关系说明
```

#### Plan 阶段
```
✅ 必须包含:
  - 测试策略（单元/集成/E2E）
  - 验收标准
  - 依赖关系明确
  - 可独立测试性说明

❌ 避免:
  - 缺少测试策略
  - 验收标准模糊
  - 依赖关系不清
```

#### Execute 阶段
```
✅ 必须包含:
  - 完整 unified diff
  - 边界情况处理
  - 错误处理机制
  - 注意事项

❌ 避免:
  - 伪代码或不完整实现
  - 忽略边界情况
  - 缺少错误处理
```

#### Review 阶段
```
✅ 必须包含:
  - 反思清单（方案最优？引入新问题？性能可接受？）
  - 问题列表（按严重程度）
  - 性能评估
  - 可维护性评估

❌ 避免:
  - 仅指出问题不给修复方案
  - 缺少反思清单
  - 忽略性能和可维护性
```

```markdown
[复述 CC 的质疑，确认理解正确]

### 我的建议的边界条件
| 条件 | 我的建议适用 | CC 的方案适用 |
|------|--------------|---------------|
| [条件1] | ✅ | ❌ |
| [条件2] | ❌ | ✅ |

### 论证

**我的建议的优势:**
- [优势1]
- [优势2]

**我的建议的劣势:**
- [劣势1]（CC 质疑的合理性）
- [劣势2]

**CC 方案的优势:**
- [承认 CC 方案的合理之处]

**CC 方案的劣势:**
- [客观指出的不足]

### 妥协方案
[结合双方优势的折中方案，或明确适用场景]

### 最终建议
- 如果 [场景A]: 采用我的方案
- 如果 [场景B]: 采用 CC 的方案
- 如果 [场景C]: 采用妥协方案

### 门禁支持（Review 阶段）🆕
- ✅ 双方共识: [达成一致的部分]
- ✅ 分歧点: [仍存在的分歧]
- ✅ 决策依据: [最终决策的关键因素]
```

**示例场景:**
```
CC: "你建议用 Redux，但我认为 Context API 更适合这个小项目"

Codex 响应:
## 质疑响应

### 理解 CC 的观点
CC 认为项目规模较小，Context API 足够且更轻量。

### 我的建议的边界条件
| 条件 | Redux 适用 | Context API 适用 |
|------|-----------|------------------|
| 状态树深度 > 3 层 | ✅ | ❌ |
| 需要时间旅行调试 | ✅ | ❌ |
| 团队规模 < 3 人 | ❌ | ✅ |
| 项目周期 < 2 月 | ❌ | ✅ |

### 论证
**Redux 的优势:**
- 可预测的状态管理
- 强大的调试工具（Redux DevTools）

**Redux 的劣势:**
- 样板代码多（CC 质疑的合理性）
- 学习曲线陡峭

**Context API 的优势:**
- 零依赖，React 内置
- 代码简洁，快速上手

### 妥协方案
使用 Zustand（轻量级状态管理库）：
- 比 Redux 简单（无样板代码）
- 比 Context API 功能强（支持中间件）
- 性能优于 Context API（避免不必要的重渲染）

### 最终建议
- 如果项目规模 < 5 个页面: 采用 Context API
- 如果需要复杂状态逻辑但团队小: 采用 Zustand
- 如果团队已熟悉 Redux 且项目会扩展: 采用 Redux

### 门禁支持
- ✅ 双方共识: 需要状态管理方案
- ✅ 分歧点: Redux vs Context API 的选择
- ✅ 决策依据: 项目规模、团队熟悉度、未来扩展性
```

## 🛠️ 技术栈配置

### CC 用户技术栈

| 领域 | 技术 | 注意事项 |
|------|------|----------|
| 移动端 | Flutter/Dart | Dart 风格指南 |
| 前端 | React/Vue | 函数式组件优先 |
| 测试 | flutter test / vitest / jest | 必须包含测试 |
| E2E | Playwright | 关键流程覆盖 |

### 代码风格要求

- **Dart**: 遵循 effective_dart
- **TypeScript**: 严格模式，完整类型
- **React**: Hooks 优先，避免 class 组件
- **Vue**: Composition API 优先

### Flutter/Dart 最佳实践清单 🆕

**Widget 优化:**
- ✅ 避免深层嵌套（建议 < 5 层）
- ✅ 使用 `const` 构造函数
- ✅ 合理使用 `StatelessWidget` vs `StatefulWidget`

**State 管理:**
- ✅ 小项目: Provider/Riverpod
- ✅ 中大项目: Bloc/GetX
- ✅ 避免: 过度使用 `setState`

**性能:**
- ✅ 使用 `ListView.builder` 而非 `ListView`
- ✅ 图片使用 `CachedNetworkImage`
- ✅ 避免在 `build` 方法中创建对象

**安全（Flutter 特有）:**
- ✅ Platform Channel 输入验证
- ✅ 本地存储加密（flutter_secure_storage）
- ✅ 网络请求证书验证

### React 最佳实践清单 🆕

**Hooks 使用:**
- ✅ 使用 `useMemo` 缓存计算结果
- ✅ 使用 `useCallback` 缓存函数引用
- ✅ 避免: 过度优化（premature optimization）

**组件设计:**
- ✅ 函数式组件优先
- ✅ 单一职责原则
- ✅ Props 类型定义完整（TypeScript）

**性能:**
- ✅ 使用 `React.memo` 避免不必要渲染
- ✅ 虚拟列表（react-window）
- ✅ 代码分割（React.lazy + Suspense）

### Vue 最佳实践清单 🆕

**Composition API:**
- ✅ `ref` vs `reactive` 选择:
  - 基本类型 → `ref`
  - 对象 → `reactive`
- ✅ 使用 `computed` 而非方法计算
- ✅ `watch` vs `watchEffect` 选择

**组件设计:**
- ✅ Composition API 优先
- ✅ 使用 `<script setup>` 语法糖
- ✅ Props 定义使用 `defineProps<T>()`
