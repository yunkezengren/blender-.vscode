# `interface_query.cc` 单文件修复：为什么有效，为什么需要，能否简化

对应文件：

- `source/blender/editors/interface/interface_query.cc`

当前单文件改动的核心代码是这两块：

```cpp
static bool button_text_is_list_item_overlay(const Button *but)
{
  BLI_assert(but->type == ButtonType::Text);

  if (but->block == nullptr) {
    return false;
  }

  for (const Button &other : but->block->buttons()) {
    if (&other == but) {
      continue;
    }
    if (!ELEM(other.type, ButtonType::ListRow, ButtonType::ViewItem)) {
      continue;
    }
    if (other.rect.xmin <= but->rect.xmin && other.rect.xmax >= but->rect.xmax &&
        other.rect.ymin <= but->rect.ymin && other.rect.ymax >= but->rect.ymax)
    {
      return true;
    }
  }

  return false;
}
```

以及：

```cpp
if ((but->type == ButtonType::Text) &&
    ELEM(but->emboss, EmbossType::None, EmbossType::NoneOrStatus) && !labeledit &&
    !(for_tooltip && !button_text_is_list_item_overlay(but)))
{
  return false;
}
```

---

## 一、为什么需要这个修改

### 原始问题

最早的规则是：

```cpp
if ((but->type == ButtonType::Text) &&
    ELEM(but->emboss, EmbossType::None, EmbossType::NoneOrStatus) && !labeledit)
{
  return false;
}
```

这表示：

- 对 `Text + None` 这种“像 label 的文字按钮”
- 如果当前不是 `Ctrl` 的 label-edit 语义
- 那它不算 interactive，不参与命中

这个规则原本是合理的，因为它保证了：

- 普通 hover / 普通点击 不会把这类 overlay text 当成正常文本框
- `Ctrl` 时才进入“可编辑 label”路径

### 后来为了 node socket rename 做的放宽

为了让 node socket name 在不按 `Ctrl` 时也能普通悬浮显示 tooltip，我们把规则放宽成了：

```cpp
... && !labeledit && !for_tooltip
```

也就是：

- 普通交互下仍然不 interactive
- 但如果这次查找是 `for_tooltip = true`，那就允许命中

这对 node socket rename 是对的，因为它解决了：

- 以前普通 hover 不显示 tooltip
- 按住 `Ctrl` hover 才显示 tooltip

### 但这个放宽范围太大

问题在于，这条规则不是只作用在 node socket rename 上，而是作用在：

- **所有 `Text + None` 按钮**

这就把列表视图里的 overlay text 一起放开了。

列表项原本的结构一般是：

- 底层一个 `ListRow` 或 `ViewItem`
- 上面叠着一个 `Text + None` 的文字按钮

原本普通 hover 命中的是：

- `ListRow` / `ViewItem`

于是 tooltip 显示的是：

- `.active_index`
- “选择列表项目”

而按住 `Ctrl` 时，`ListRow` 会被排掉，命中的才是 overlay text，于是 tooltip 变成：

- `.name`

把 `for_tooltip` 全局放开以后，普通 hover 也直接命中 overlay text 了，于是：

- 普通 hover 不再显示 `.active_index`
- 而是直接显示 `.name`

同时 hover 目标在 `ListRow/ViewItem` 和 overlay `Text` 之间跳来跳去，还会导致：

- 鼠标移动时文字忽明忽暗

所以这次单文件修改的必要性就在这里：

- 既要保住 node socket rename 的普通 hover tooltip
- 又要把 list/view item 的原有 hover 语义保回来

---

## 二、为什么这个单文件修改有效

### 关键思路

核心思路不是再去区分“node socket vs list item”，而是：

- **只在 `interface_query.cc` 这一层，识别当前这颗 `Text + None` 是否其实是某个列表项上面的 overlay text**

如果是，就沿用旧行为。  
如果不是，就允许普通 hover tooltip。

### `button_text_is_list_item_overlay()` 在干什么

```cpp
static bool button_text_is_list_item_overlay(const Button *but)
```

这个 helper 只接受：

- `ButtonType::Text`

然后做三步判断。

#### 1. 必须挂在某个 block 上

```cpp
if (but->block == nullptr) {
  return false;
}
```

因为后面要去同一个 block 里找“覆盖它的列表项按钮”。

#### 2. 在同一个 block 里找 `ListRow` 或 `ViewItem`

```cpp
for (const Button &other : but->block->buttons()) {
  ...
  if (!ELEM(other.type, ButtonType::ListRow, ButtonType::ViewItem)) {
    continue;
  }
```

这一步抓的是：

- 列表项本体
- 树视图项本体

因为真正的“列表 hover 语义”在它们身上，不在覆盖文字按钮身上。

#### 3. 看这个 `Text` 是否被它完整包住

```cpp
if (other.rect.xmin <= but->rect.xmin && other.rect.xmax >= but->rect.xmax &&
    other.rect.ymin <= but->rect.ymin && other.rect.ymax >= but->rect.ymax)
{
  return true;
}
```

这一步是整个修复能成立的关键。

它在判断：

- 当前这颗 `Text`
- 是不是几何上完全落在某个 `ListRow` / `ViewItem` 的矩形内部

如果是，就把它当成：

- “列表项上面的 overlay text”

为什么这个启发式有效？

因为列表视图的典型结构就是：

- row/item 负责 hover、选择、tooltip 语义
- text 按钮只是叠在它上面，负责名字显示和 rename 输入

所以“被 `ListRow/ViewItem` 完整包住的 `Text + None`”正好很像我们要排除的对象。

### 最后的条件为什么能同时满足两个目标

原来的放宽是：

```cpp
!(for_tooltip)
```

现在改成：

```cpp
!(for_tooltip && !button_text_is_list_item_overlay(but))
```

把它展开理解就是：

- 普通情况下，`Text + None` 仍然不 interactive
- 只有当：
  - 当前是 tooltip/hover 查找
  - 并且这颗文字按钮不是 list item overlay
  - 才允许命中

这会带来两个结果：

#### 对 node socket rename

它是 `Text + None`，但不是盖在 `ListRow/ViewItem` 上的 overlay text。  
所以：

- `button_text_is_list_item_overlay(but) == false`
- 普通 hover tooltip 被放开

于是 node socket rename 又能直接悬浮显示 tooltip。

#### 对 list/view item overlay text

它确实是盖在 `ListRow/ViewItem` 上的。  
所以：

- `button_text_is_list_item_overlay(but) == true`
- 普通 hover tooltip 不再放开

于是命中逻辑回到旧行为：

- 普通 hover 仍然优先落到 `ListRow/ViewItem`
- Ctrl-hover 才会落到 overlay text

这就是为什么它能同时修住两个问题。

---

## 三、为什么它还能修掉“文字忽明忽暗”

这个现象不是单独的绘制 bug，而是 hover 目标在抖。

放宽前：

- 普通 hover 主要命中 `ListRow` / `ViewItem`

放宽后：

- 鼠标稍微移动，可能一会命中 row/item，一会命中 overlay text

这两个按钮的：

- `UI_HOVER`
- 文字颜色
- 选中/高亮绘制

处理路径并不完全一样，所以你就会看到：

- 鼠标移动时文字颜色一会亮一会暗

这次单文件修复把 list item overlay text 从普通 hover tooltip 路径里重新排掉之后：

- 普通 hover 命中目标重新稳定回 `ListRow/ViewItem`

于是文字颜色的抖动也一起消失。

所以第二个问题并不是单独修的，而是第一个问题修好后自然恢复的。

---

## 四、为什么说这个修改是“启发式”的

它不是通过一个显式 flag 或显式类型系统在说：

- “这是 node socket rename 按钮”
- “这是 list item overlay text”

而是通过几何关系来猜：

- “一颗 `Text` 如果完全被 `ListRow/ViewItem` 包住，那它多半是覆盖文字按钮”

这就是典型的启发式修复：

- 好处：只改一个文件，改动最小
- 代价：规则不是显式语义，而是基于现有布局结构推断

为什么这版仍然有效？

因为当前碰到的问题场景里，这个推断恰好和真实结构一致。

---

## 五、能否简化

### 可以简化到什么程度

从“改动文件数”来说，这已经很简了：

- 只改 `interface_query.cc`

从“代码行数”来说，还能有一些小收缩空间，但不会本质减少复杂度。

比如可以把 helper 写得更紧凑一点，或者合并判断，但本质上仍然要做两件事：

1. 识别 overlay text
2. 只对非 overlay text 放开 `for_tooltip`

这两个步骤本身是省不掉的。

### 为什么不能再简单成“一句排除列表项”

因为走到这段逻辑时，当前按钮类型已经是：

- `ButtonType::Text`

不是：

- `ButtonType::ListRow`
- `ButtonType::ViewItem`

所以不能简单写成：

```cpp
if (but->type == Text && but->type != ListRow)
```

这种判断没有信息量。

真正的问题不是“当前按钮是不是列表项”，而是：

- “当前这颗 `Text` 是不是挂在列表项上面的 overlay text”

这就决定了必须做一层额外识别。

### 和多文件显式标记方案相比

如果用更“正统”的做法，一般会是：

- 给 node socket rename 打一个专用 flag
- query 层只对带这个 flag 的按钮放开普通 hover tooltip

那个方案更稳、更显式，但会多改几份文件。

当前这版单文件修复的取舍是：

- **接受一点启发式**
- 换来最小改动范围

所以它不是“最正统”的方案，但是“最省改动、当前有效”的方案。

---

## 六、这个修改到底在保护什么行为边界

可以把它理解成在保护这样一条边界：

- **node socket rename label**
  - 需要普通 hover tooltip

- **list/view item overlay text**
  - 不应该抢走普通 hover 的 row/item tooltip
  - 只应该在 `Ctrl` 的 label-edit 语义下参与命中

当前这版单文件修改，实际上就是把这条边界重新补回来了。

---

## 七、结论

### 为什么需要

因为把 `Text + None` 的 `for_tooltip` 全局放开以后：

- node socket tooltip 是好了
- 但 list/view item 的 overlay text 也一起被放开了

导致：

- 普通 hover tooltip 从 `.active_index` 变成 `.name`
- 文字颜色随鼠标移动抖动

### 为什么有效

因为它没有回退 node socket 的普通 hover tooltip，
而是只把“盖在 `ListRow/ViewItem` 上面的 overlay text”重新排除出去。

于是：

- node socket 继续受益
- list/view item 恢复旧行为

### 能否简化

- 从文件数上，这已经是很小的修法
- 从逻辑上，还能写得更短，但不能省掉“识别 overlay text”这一步

因为问题的关键不在“它是不是 `Text`”，而在：

- “这颗 `Text` 是独立的 label-like text，还是列表项上面的 overlay text”
