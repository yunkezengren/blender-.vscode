# 双击修改接口名：当前三处修改的调用链

## 问题

想要让这种 `Text + EmbossType::None` 的 node socket 名字，在不按 `Ctrl` 时也支持双击进入重命名，当前暂存区里的三处修改是怎么联系起来的？

## 先看目标按钮是什么

在 `source/blender/editors/space_node/drawnode.cc` 里，这类 socket name 不是普通 label，而是：

- `layout->prop(...)` 生成的 RNA 文本按钮
- `layout->emboss_set(ui::EmbossType::None)` 让它变成无边框文本按钮

所以后面各层逻辑看到的，其实是一颗：

- `but->type == ButtonType::Text`
- `but->emboss == EmbossType::None`

的按钮。

---

## 按调用顺序看三处修改

### 1. `interface_query.cc`：先让按钮在普通悬浮下也能被找到

文件：`source/blender/editors/interface/interface_query.cc`

位置：`button_is_interactive_ex(...)`

当前修改：

```cpp
if ((but->type == ButtonType::Text) &&
    ELEM(but->emboss, EmbossType::None, EmbossType::NoneOrStatus) && !labeledit &&
    !for_tooltip)
{
  return false;
}
```

这段的作用是：

- 对 `Text + None` 这种按钮，默认仍然不把它当成“普通可交互按钮”
- 但是如果当前是 `for_tooltip = true` 的鼠标悬浮查找路径，就不要太早 `return false`

这一步不是直接支持双击，而是为了让后面的 hover 激活链条先跑起来。

---

### 2. `interface_handlers.cc`：鼠标移动时先命中它，再把它标成 `disable_force`

文件：`source/blender/editors/interface/interface_handlers.cc`

在鼠标移动处理里：

```cpp
const bool labeledit = event->modifier & KM_CTRL;
const bool for_tooltip = true;
Button *but = button_find_mouse_over_ex(
    region, event->xy, labeledit, for_tooltip, nullptr, nullptr);
```

这里会调用 `button_find_mouse_over_ex(...)`，它内部会走到上一节的
`button_is_interactive_ex(...)`。

因为现在 `for_tooltip = true`，所以这颗 `Text + None` 按钮可以先被命中。

命中后会执行：

```cpp
button_activate_init(C, region, but, BUTTON_ACTIVATE_OVER);
```

这会让按钮进入 hover/highlight 状态。

但紧接着又会检查：

```cpp
if (but->active && !button_is_interactive(but, labeledit)) {
  but->active->disable_force = true;
}
```

注意这里改回了普通交互判断，也就是 `for_tooltip = false`。  
于是这颗按钮虽然已经 hover 住了，但仍然会被标成：

- `but->active != nullptr`
- `data->state == BUTTON_STATE_HIGHLIGHT`
- `data->disable_force == true`

这就是后面双击要接住的状态。

---

### 3. `interface_handlers.cc:8864`：在 `disable_force` 状态里专门放行 `Ctrl+Click / Double-Click`

文件：`source/blender/editors/interface/interface_handlers.cc`

当前修改：

```cpp
if (data->disable_force && but->type == ButtonType::Text &&
    ELEM(but->emboss, EmbossType::None, EmbossType::NoneOrStatus) &&
    event->type == LEFTMOUSE && (event->modifier & KM_CTRL || event->val == KM_DBL_CLICK))
{
  button_activate_state(C, but, BUTTON_STATE_TEXT_EDITING);
  return WM_UI_HANDLER_BREAK;
}
```

这段的作用是：

- 这颗按钮已经被 hover 到了
- 但它目前仍处在 `disable_force` 状态，不能按普通文本框那样自然进入编辑
- 所以这里给了两个例外入口：
  - `Ctrl+Click`
  - `Double-Click`

只要满足这个条件，就直接把按钮切进：

```cpp
BUTTON_STATE_TEXT_EDITING
```

如果没有这一处，那么这颗按钮虽然能 hover、高亮，但双击仍可能被“它不是普通 interactive button”这件事挡掉。

---

### 4. `interface_handlers.cc:5220`：文本按钮自己的逻辑也要正式承认双击

文件：`source/blender/editors/interface/interface_handlers.cc`

函数：`do_but_TEX(...)`

当前修改：

```cpp
const bool is_double_click = event->type == LEFTMOUSE && event->val == KM_DBL_CLICK;
if ((ELEM(event->type, LEFTMOUSE, EVT_BUT_OPEN, EVT_PADENTER, EVT_RETKEY) &&
     event->val == KM_PRESS) ||
    is_double_click)
{
  ...
  else if (ELEM(but->emboss, EmbossType::None, EmbossType::NoneOrStatus) &&
           (event->modifier != KM_CTRL) && !is_double_click)
  {
    /* Pass. */
  }
  else {
    button_activate_state(C, but, BUTTON_STATE_TEXT_EDITING);
  }
}
```

这一步的意义是：

- 以前这里只把普通 `KM_PRESS` 当成进入文本编辑的入口
- 对 `Text + None` 按钮来说，没按 `Ctrl` 时还会 `Pass`
- 改完后，`KM_DBL_CLICK` 也被当成合法入口
- 并且对“没按 `Ctrl` 就 `Pass`”这条规则，双击被排除在外

所以这里是在文本按钮自己的处理逻辑里，正式承认“双击也可以进入文本编辑”。

---

## 三处修改如何串起来

按时序压缩成一条链就是：

1. 鼠标移动时，`button_find_mouse_over_ex(...)` 要先能找到这颗 `Text + None` 按钮  
   这由 `interface_query.cc` 那处 `!for_tooltip` 改动保证。

2. 找到后，按钮被激活成 hover/highlight  
   但因为它本质上仍不是普通 interactive 文本框，所以会被标成 `disable_force = true`。

3. 用户双击时，`interface_handlers.cc:8864` 那处特判在 `disable_force` 状态里专门放行  
   直接切进 `BUTTON_STATE_TEXT_EDITING`。

4. 真正的文本按钮处理逻辑 `do_but_TEX(...)` 也必须认可 `KM_DBL_CLICK`  
   这由 `interface_handlers.cc:5220` 那处改动保证。

所以三处分工可以概括成：

- `interface_query.cc`：先让 hover 命中成立
- `interface_handlers.cc:8864`：在 `disable_force` 状态里给双击开门
- `interface_handlers.cc:5220`：让文本编辑逻辑本身承认双击入口

---

## 这三处能不能少

按当前实现思路，不建议少。

- 少 `interface_query.cc` 这处，最容易先坏掉的是“不按 Ctrl 时，鼠标普通悬浮根本命不中这颗按钮”，那后面的双击放行也没有前提。
- 少 `interface_handlers.cc:8864` 这处，按钮即使 hover 到了，也可能仍被 `disable_force` 拦住。
- 少 `interface_handlers.cc:5220` 这处，文本按钮自己的逻辑还没把双击当成正式入口。

所以在当前这套实现下，这三处分别守着不同关口，不是简单重复。

---

## `button_find_mouse_over_ex()` / `but_find_mouse_over()` 是干什么的

文件：`source/blender/editors/interface/interface_query.cc:332~388`

### 一句话版本

它们是“在当前鼠标位置，找出鼠标正压着哪一颗 UI 按钮”的入口函数。

也可以把它理解成：  
“把窗口坐标里的鼠标位置，映射到当前 region 里的所有 UI blocks / buttons 上，然后返回最上层、最符合条件的那颗按钮。”

### `button_find_mouse_over_ex(...)` 做了什么

函数签名：

```cpp
Button *button_find_mouse_over_ex(const ARegion *region,
                                  const int xy[2],
                                  const bool labeledit,
                                  const bool for_tooltip,
                                  const ButtonFindPollFn find_poll,
                                  const void *find_custom_data)
```

它的职责大致是：

1. 先检查鼠标点是不是在这个 `region` 里  
   如果不在，直接返回 `nullptr`。

2. 遍历这个 region 里的所有 `uiblocks`

```cpp
for (Block &block : region->runtime->uiblocks)
```

3. 把窗口坐标的鼠标位置换算到当前 block 的局部坐标里

```cpp
window_to_block_fl(region, &block, &mx, &my);
```

4. 跳过一些特殊情况  
   比如菜单自动滚动条区域：

```cpp
if ((block.flag & BLOCK_CLIPTOP && block.rect.ymax - UI_MENU_SCROLL_MOUSE < my) ||
    (block.flag & BLOCK_CLIPBOTTOM && block.rect.ymin + UI_MENU_SCROLL_MOUSE > my))
{
  continue;
}
```

5. 倒序遍历 block 里的按钮

```cpp
for (Button &but : block.buttons() | std::views::reverse)
```

这里倒序很重要，因为通常后画出来、视觉上更靠上的按钮，也应该先被命中。

6. 对每颗按钮先做资格判断

```cpp
if (button_is_interactive_ex(&but, labeledit, for_tooltip))
```

这一步决定“这颗按钮在当前语义下算不算能被命中的候选按钮”。

这里的两个参数很关键：

- `labeledit`
  - 一般表示当前是不是按着 `Ctrl`
  - 会影响某些“平时像 label，按 Ctrl 才可编辑”的按钮是否算 interactive

- `for_tooltip`
  - 表示这次查找是不是为了 hover / tooltip / over 状态
  - 它允许某些按钮“虽然不是正常点击目标，但 hover 时仍然可以先找到它”

7. 真正做几何命中测试

普通按钮走：

```cpp
button_contains_pt(&but, mx, my)
```

饼菜单按钮走：

```cpp
but_isect_pie_seg(&block, &but)
```

8. 一旦找到命中的按钮，就把它记到 `butover`，并结束当前 block 的查找。

9. 如果 block 带 `BLOCK_CLIP_EVENTS`，并且鼠标点已经落在它的矩形范围里，就不要再把事件继续往后面的 block 传了。

```cpp
if (block.flag & BLOCK_CLIP_EVENTS) {
  if (BLI_rctf_isect_pt(&block.rect, mx, my)) {
    break;
  }
}
```

10. 最后返回找到的那颗按钮；找不到就返回 `nullptr`。

### `but_find_mouse_over(...)` 是什么

它只是一个很薄的包装：

```cpp
Button *but_find_mouse_over(const ARegion *region, const wmEvent *event)
{
  return button_find_mouse_over_ex(
      region, event->xy, event->modifier & KM_CTRL, false, nullptr, nullptr);
}
```

也就是说它只是把：

- 鼠标坐标 `event->xy`
- 当前是否按着 `Ctrl`
- `for_tooltip = false`

这几个常见参数帮你填好，然后转调 `button_find_mouse_over_ex(...)`。

所以：

- `button_find_mouse_over_ex(...)` 是完整版、可定制版
- `but_find_mouse_over(...)` 是常用默认版

### 它和双击重命名这件事的关系

双击重命名那条链，第一步就依赖“鼠标当前到底压着哪颗按钮”。

而这个判断正是 `button_find_mouse_over_ex(...)` 做的。

这也是为什么前面 `interface_query.cc` 那处 `!for_tooltip` 改动会有作用：  
它改变的就是这颗 `Text + None` 按钮，在 `button_find_mouse_over_ex(..., for_tooltip=true)` 这条 hover 查找路径里，到底能不能先被找到。
