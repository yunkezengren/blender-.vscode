# 双击修改接口名：命名、状态和调用细节

## 1. `disable_force` 是干什么的？为什么这样命名？

相关位置：

- `source/blender/editors/interface/interface_handlers.cc:444~449`
- `source/blender/editors/interface/interface_handlers.cc:10003`

结构体里已经有注释：

```cpp
/**
 * Behave as if #BUT_DISABLED is set (without drawing grayed out).
 * Needed so non-interactive labels can be activated for the purpose of showing tool-tips,
 * without them blocking interaction with nodes, see: #97386.
 */
bool disable_force = false;
```

它的作用可以直接概括成：

- **逻辑上像 disabled**
- **视觉上不画成灰掉**

也就是说，它不是按钮真正带了 `BUT_DISABLED` 标志，而是运行时临时“强制按 disabled 的方式处理”。

在 `do_button()` 里能看到它怎么生效：

```cpp
const bool is_disabled = but->flag & BUT_DISABLED || data->disable_force;
```

所以 `disable_force` 的语义就是：

- `BUT_DISABLED` 是按钮本体静态配置上的 disabled
- `disable_force` 是事件处理阶段附加上的“临时强制 disabled”

之所以叫 `disable_force`，就是因为它不是按钮天生 disabled，而是处理逻辑为了某个目的，**强行把它当 disabled**。

这里那个“某个目的”，就是：

- 这颗按钮可以先被 hover / active 到，用来显示 tooltip、hover 状态
- 但又不想让它像真正可编辑文本框一样吞掉普通点击，影响 node 本体交互

---

## 2. 为什么命名以 `over` 结尾？

相关位置：

- `source/blender/editors/interface/interface_handlers.cc:9983`
- `source/blender/editors/interface/interface_handlers.cc:9991`

这里的 `over` 是 UI 事件系统里一个很常见的命名，意思就是：

- mouse over
- hover over
- 光标当前“压在……上面”

比如：

- `handle_button_over(...)`
- `button_find_mouse_over_ex(...)`
- `BUTTON_ACTIVATE_OVER`

它们都不是“点击”语义，而是“鼠标移动到某个按钮上方”的语义。

所以这里叫 `over`，是因为这条逻辑处理的是：

- 当前鼠标悬浮在哪颗按钮上
- 哪颗按钮要进入 hover / highlight 状态

而不是“按下后选中谁”。

## 3. 为什么变量名叫 `labeledit` 和 `for_tooltip`？

相关位置：

- `source/blender/editors/interface/interface_handlers.cc:9986`

### `labeledit`

```cpp
const bool labeledit = event->modifier & KM_CTRL;
```

这个名字比较老，字面上可以理解成：

- “当前是不是以 label-edit 的语义在查找/处理按钮”

它最早服务的是这类需求：

- 某些东西平时像 label
- 但按住 `Ctrl` 后，可以把它当成“可编辑 label”来处理

所以它不等于“按钮已经在 edit”，而更接近：

- “当前输入语义是不是允许 label 风格的东西进入编辑路径”

在 node socket rename 这个场景里，`Ctrl` 就是打开这条 label-edit 语义的开关。

### `for_tooltip`

```cpp
const bool for_tooltip = true;
```

这个名字更直接，意思就是：

- 这次命中查找是不是为了 tooltip / hover 目的

因为 UI 里有一类控件：

- 平时不应该算正常可点击目标
- 但 hover 时仍然需要先命中它，才能显示 tooltip 或 hover 效果

所以 `for_tooltip` 的真正语义其实是：

- “这次是不是为了 hover/tooltip 这类较宽松的查找”

不只是 tooltip 文本本身，还包括：

- hover 激活
- 高亮
- tooltip 展示前提

## 4. 这四个函数分别是干什么的？

这里的四个函数是：

- `button_find_mouse_over_ex(...)`
- `but_find_mouse_over(...)`
- `button_is_interactive_ex(...)`
- `button_is_interactive(...)`

### `button_find_mouse_over_ex(...)`

文件：`source/blender/editors/interface/interface_query.cc:332`

这是完整版的“按鼠标位置找按钮”函数。

它做的事是：

1. 检查鼠标是否在当前 region 内
2. 遍历 region 里的所有 `uiblocks`
3. 把窗口坐标换算到每个 block 的局部坐标
4. 倒序遍历 block 里的按钮
5. 先用 `button_is_interactive_ex(...)` 判断这颗按钮在当前语义下是不是候选者
6. 再做几何命中测试，判断鼠标是否真的压在这颗按钮上
7. 返回找到的最上层那颗按钮

它是最通用、最底层的入口。

### `but_find_mouse_over(...)`

文件：`source/blender/editors/interface/interface_query.cc:377`

它只是一个薄封装：

```cpp
return button_find_mouse_over_ex(
    region, event->xy, event->modifier & KM_CTRL, false, nullptr, nullptr);
```

也就是帮你自动填好：

- 鼠标坐标
- 当前是否按着 `Ctrl`
- `for_tooltip = false`

所以它是“默认版鼠标命中查找”。

### `button_is_interactive_ex(...)`

文件：`source/blender/editors/interface/interface_query.cc:73`

这是完整版的“这颗按钮当前算不算 interactive”判断。

它会考虑：

- 按钮类型
- 是否隐藏 / 滚动裁剪
- 当前是不是 `label-edit` 语义
- 当前是不是 tooltip / hover 查找语义

也就是说，它不是简单看 `BUT_DISABLED`，而是在问：

- “在这次具体查找/交互语义下，这颗按钮应不应该参与命中和响应？”

### `button_is_interactive(...)`

文件：`source/blender/editors/interface/interface_query.cc:123`

它也是一个薄封装：

```cpp
return button_is_interactive_ex(but, labeledit, false);
```

也就是默认按“正常交互”来判断，而不是 tooltip / hover 宽松模式。

### 两对 `_ex` 的含义

Blender 里很多 `_ex` 命名都表示：

- 基础版：常用默认参数
- `_ex` 版：额外参数更多、更通用

所以这里：

- `button_find_mouse_over_ex(...)` 是扩展版
- `but_find_mouse_over(...)` 是便捷版
- `button_is_interactive_ex(...)` 是扩展版
- `button_is_interactive(...)` 是便捷版

## 5. 为什么这段逻辑放在 `do_button()` 这里？

相关位置：

- `source/blender/editors/interface/interface_handlers.cc:8866~8873`

当前代码：

```cpp
if (data->disable_force && but->type == ButtonType::Text &&
    ELEM(but->emboss, EmbossType::None, EmbossType::NoneOrStatus) &&
    event->type == LEFTMOUSE && (event->modifier & KM_CTRL || event->val == KM_DBL_CLICK))
{
  button_activate_state(C, but, BUTTON_STATE_TEXT_EDITING);
  return WM_UI_HANDLER_BREAK;
}
```

### 为什么放这里

因为它处理的是一个**非常早的拦截点**：

```cpp
const bool is_disabled = but->flag & BUT_DISABLED || data->disable_force;
...
if (is_disabled) {
  return WM_UI_HANDLER_CONTINUE;
}
```

也就是说，如果不在这个位置提前特判：

- 这颗按钮一旦因为 `disable_force` 被当成 disabled
- 后面的正常文本按钮处理逻辑根本走不到

所以它必须放在：

- `is_disabled` 早退之前
- `do_but_TEX(...)` 之上

它的职责就是：

- 给这类“逻辑上被强制 disabled 的 hover 按钮”开一个特例入口

### 为什么这里还要 `event->modifier & KM_CTRL`

因为这段不只是为双击补洞，也是在**保住已有的 Ctrl+单击语义**。

之前之所以 Ctrl+左键单击已经能修改，是因为：

- `Ctrl` 会让 `labeledit = true`
- 这颗按钮在正常交互判断里就不再被排除
- 于是能走到后面的文本编辑逻辑

但现在有了一条 `disable_force` 路径以后，某些时候事件可能先卡在这个“逻辑 disabled”分支前面。  
这里把 `Ctrl+Click` 也纳进来，是为了确保：

- 老的 `Ctrl+Click` 行为不回退
- 新的 `Double-Click` 行为也能走同一个后门

如果这里只写双击，不写 `Ctrl`，就会把“已有行为”和“新行为”拆成两条不完全一致的路径，后续更容易出现回归。

### `WM_UI_HANDLER_BREAK` 是干什么的

它的意思是：

- 当前 UI handler 已经把这个事件消费掉了
- 事件不要再继续往别的 UI / 区域 / 处理逻辑传播

可以把它理解成：

- “这事我接管了，到此为止”

和它相对的是：

- `WM_UI_HANDLER_CONTINUE`

表示：

- 当前逻辑没真正消费这个事件，后面还能继续处理

### `button_activate_state(C, but, BUTTON_STATE_TEXT_EDITING);` 是干什么的

它是在切按钮的交互状态机。

这里的意思就是：

- 把当前按钮从 hover/highlight 状态
- 切换到文本编辑状态

进入这个状态后，后面才会：

- 建立文本编辑缓冲
- 显示光标 / 选区
- 让键盘输入落到这个文本按钮上

### “之前没设置，不也能 Ctrl+左键单击修改吗？”

对，之前能。  
但那是靠**正常的 Ctrl+label-edit 路径**做到的，不是靠这里。

这里新增的意义是：

- 对 `disable_force` 这条特殊路径，补一个显式后门
- 让 `Ctrl+Click` 和 `Double-Click` 在这里都可靠成立

所以它不是“从无到有发明 Ctrl+Click”，而是：

- 把已有 Ctrl+Click 语义也覆盖到这条特殊分支里
- 同时顺手把 Double-Click 放进去

## 6. `BUTTON_STATE_HIGHLIGHT` 是什么？为什么加了 `is_double_click` / `!is_double_click` 就行了？

相关位置：

- `source/blender/editors/interface/interface_handlers.cc:5222~5232`

### `BUTTON_STATE_HIGHLIGHT` 的作用

它是按钮已经被激活、但还没有真正进入“编辑/拖拽/应用”等更深状态之前的基础 hover/active 状态。

可以把它理解成：

- 鼠标当前已经在这颗按钮上
- 这颗按钮是当前活跃的 UI 目标
- 接下来收到点击、按键等输入时，就由它继续决定往哪个更具体的状态走

在 `button_activate_init(..., BUTTON_ACTIVATE_OVER)` 里也能看到：

```cpp
button_activate_state(C, but, BUTTON_STATE_HIGHLIGHT);
```

所以 `HIGHLIGHT` 实际上就是“over/hover 激活后”的基础状态。

### 为什么加两个 `is_double_click` / `!is_double_click` 就解决了

原来的 `do_but_TEX(...)` 逻辑是：

1. 只接受普通 `KM_PRESS`
2. 对 `Text + None` 按钮，如果没按 `Ctrl`，就 `Pass`

这正好把双击挡住了两次：

- `KM_DBL_CLICK` 根本不被视为合法入口
- 即使继续看，`Text + None` 且没按 `Ctrl` 仍然会 `Pass`

改动后：

```cpp
const bool is_double_click = event->type == LEFTMOUSE && event->val == KM_DBL_CLICK;
```

第一处改动：

```cpp
... || is_double_click
```

让双击被纳入“合法触发事件”。

第二处改动：

```cpp
(event->modifier != KM_CTRL) && !is_double_click
```

让“无 Ctrl 则 Pass”这条规则对双击失效。

所以它看起来只是加了两个条件，但实际上刚好移除了双击的两个阻塞点。

## 7. 对比这两个条件，它们分别在管什么？

第一处，在 `do_but_TEX(...)` 里：

```cpp
ELEM(but->emboss, EmbossType::None, EmbossType::NoneOrStatus) &&
(event->modifier != KM_CTRL) && !is_double_click
```

第二处，在 `do_button()` 里：

```cpp
data->disable_force && but->type == ButtonType::Text &&
ELEM(but->emboss, EmbossType::None, EmbossType::NoneOrStatus) &&
event->type == LEFTMOUSE && (event->modifier & KM_CTRL || event->val == KM_DBL_CLICK)
```

它们看起来都在判断 `Text + None`，但职责不同。

### 第一处：文本按钮自己的“拒绝规则”

它位于 `do_but_TEX(...)` 内部，语义是：

- 这已经是一颗文本按钮了
- 现在决定要不要进入文本编辑
- 对无边框文本按钮，普通单击且没按 `Ctrl` 时，不进编辑

所以这是一个**文本编辑入口过滤器**。

改完后它的意思变成：

- 普通无修饰单击：还是不进
- `Ctrl+Click`：可以进
- `Double-Click`：也可以进

### 第二处：`disable_force` 下的“提前放行规则”

它位于 `do_button()` 更外层，语义是：

- 这颗按钮当前已经被 hover/active 到了
- 但又被 `disable_force` 临时当成 disabled
- 正常情况下会在 `is_disabled` 处提前返回

所以这里不是在做“最终文本编辑判定”，而是在做：

- **在被强制 disabled 的特殊情况下，先开一道门，让这类事件别被提前吞没**

### 对比一句话总结

第一处是：

- “到了文本按钮逻辑里，哪些事件算合法进入编辑”

第二处是：

- “在还没走到文本按钮逻辑之前，哪些事件有资格穿过 `disable_force` 这层拦截”

也就是：

- 第二处解决“你能不能走到门口”
- 第一处解决“到了门口之后让不让你进去”
