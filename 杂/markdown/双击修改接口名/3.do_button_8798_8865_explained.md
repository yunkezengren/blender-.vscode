# `do_button()` 8798~8865 代码详解

对应代码位置：

- `source/blender/editors/interface/interface_handlers.cc:8798~8865`

这段代码是 `do_button(...)` 的开头部分。  
它处理的是：

- 这颗按钮已经是当前 active button
- 现在又收到一个新事件
- 在真正进入各按钮类型的细分处理之前，先统一处理一批“高优先级通用逻辑”

可以把它理解成：

- **按钮通用事件分发的前置层**

---

## 先看函数入口

```cpp
static int do_button(bContext *C, Block *block, Button *but, const wmEvent *event)
{
  HandleButtonData *data = but->active;
  int retval = WM_UI_HANDLER_CONTINUE;

  const bool is_disabled = but->flag & BUT_DISABLED || data->disable_force;
```

### `HandleButtonData *data = but->active;`

`but->active` 是这颗按钮当前交互期挂着的运行时状态。

里面会放很多“只在激活期间有意义”的东西，比如：

- 当前状态机状态 `data->state`
- `disable_force`
- `tooltip_force`
- 文本编辑相关缓存
- 当前 region/window

所以 `do_button()` 一进来先拿到 `data`，后面所有交互判断都要用它。

### `retval = WM_UI_HANDLER_CONTINUE`

先假定：

- 这个事件还没有被当前按钮消费
- 后续如果没有命中任何处理逻辑，就继续往外传

后面如果某条分支真的处理了事件，通常就会返回：

- `WM_UI_HANDLER_BREAK`

表示“事件到此为止，被我吃掉了”。

### `is_disabled`

```cpp
const bool is_disabled = but->flag & BUT_DISABLED || data->disable_force;
```

这里把两种“disabled 语义”统一起来：

1. `but->flag & BUT_DISABLED`
   - 按钮本体就是真 disabled

2. `data->disable_force`
   - 事件系统临时把它当 disabled
   - 视觉上不一定灰掉

所以从这往后看，很多逻辑不在乎它是哪一种 disabled，只关心：

- 当前是不是应该像 disabled button 一样对待

---

## 断言：pointype 和 poin 必须成对

```cpp
/* If `but->pointype` is set, `but->poin` should be too. */
BLI_assert(!bool(but->pointype) || but->poin);
```

这是一条内部一致性检查。

意思是：

- 如果按钮声明了 `pointype`
- 那它也必须真的有 `poin`

也就是：

- 不能只说“我指向一种类型”
- 却没有实际指针数据

这不是业务逻辑，而是防止内部状态损坏。

---

## 这里处理的是“硬编码”的通用按钮交互

```cpp
/* Only hard-coded stuff here, button interactions with configurable
 * keymaps are handled using operators (see #keymap_ui). */
```

这句注释很重要。

它在说：

- `do_button()` 这里处理的是 UI 系统内部固定写死的交互
- 可配置 keymap 的那类行为，不在这里直接写分支，而是走 operator / keymap 系统

所以你会看到下面这一段主要都是：

- copy/paste
- 右键菜单
- disabled 特判
- 双击 / Ctrl+Click 之类

这些都属于按钮系统自己掌握的基础行为。

---

## 只在 `BUTTON_STATE_HIGHLIGHT` 下走这整段逻辑

```cpp
if (data->state == BUTTON_STATE_HIGHLIGHT) {
```

这表示这整段代码只处理一种前提状态：

- 这颗按钮已经被 hover/activate 到了
- 但还没有进入更深层状态，比如：
  - 文本编辑
  - 数值编辑
  - 拖拽
  - 菜单展开

也就是说，这里是：

- **hover/highlight 阶段收到输入后怎么办**

这很合理，因为下面马上要处理的都是：

- 点一下
- 右键
- Ctrl+C / Ctrl+V
- disabled 特例

这些通常都发生在“按钮已经是当前 hover/active 目标”之后。

---

## Ctrl+C / Ctrl+V 的统一预判

```cpp
bool is_press_ctrl_but_no_shift = (event->val == KM_PRESS) &&
                                  (event->modifier & (KM_CTRL | KM_OSKEY)) &&
                                  (event->modifier & KM_SHIFT) == 0;
const bool do_copy = event->type == EVT_CKEY && is_press_ctrl_but_no_shift;
const bool do_paste = event->type == EVT_VKEY && is_press_ctrl_but_no_shift;
```

这一段是在做三个层次的拆分。

### 第一层：是不是“按下 Ctrl/OSKey 且没按 Shift”

```cpp
is_press_ctrl_but_no_shift
```

它要求：

- 事件是 `KM_PRESS`
- 修饰键里有 `Ctrl` 或 `OSKey`
- 没有 `Shift`

这里把 macOS 的 `Command` 也兼容进来了，所以用了：

- `KM_CTRL | KM_OSKEY`

### 第二层：是不是 copy

```cpp
const bool do_copy = event->type == EVT_CKEY && is_press_ctrl_but_no_shift;
```

也就是：

- `Ctrl+C`

### 第三层：是不是 paste

```cpp
const bool do_paste = event->type == EVT_VKEY && is_press_ctrl_but_no_shift;
```

也就是：

- `Ctrl+V`

之所以先在这里统一算出来，是因为下面几处逻辑都会复用：

- list row 特判
- disabled 时允许 copy
- 后续 paste 处理

## ListRow 上的 copy/paste 转发到重叠的文本按钮

```cpp
/* Specific handling for list-rows, we try to find their overlapping text button. */
if ((do_copy || do_paste) && but->type == ButtonType::ListRow) {
  Button *labelbut = but_list_row_text_activate(C, but, data, event, BUTTON_ACTIVATE_OVER);
  if (labelbut) {
    but = labelbut;
    data = but->active;
  }
}
```

这一段是在解决一个 UI 结构问题：

- 有些界面里，当前 active 的是一个整行 `ListRow`
- 但真正承载文本内容、也真正该响应 copy/paste 的，是叠在上面的文字按钮

所以这里做的是一次“目标重定向”：

1. 如果当前事件是 copy/paste
2. 且当前按钮是 `ListRow`
3. 就尝试找到它覆盖的文本按钮 `labelbut`
4. 如果找到了，就把后续处理对象从原来的 `but` 切成 `labelbut`

```cpp
but = labelbut;
data = but->active;
```

这两句很关键，表示从这里往后：

- 后面的逻辑都以文本按钮为准
- 不再按整行 row 来处理

可以把它理解成：

- “copy/paste 时，别作用在 row 容器上，作用到真正的文本内容上”

---

## disabled 按钮盖在 list row 上时，允许点到底下那一行

```cpp
/* When clicking on a disabled button on top of a list row, activate the list row below it.
 * See #150341. */
if (is_disabled && (event->type == LEFTMOUSE) && (event->val == KM_PRESS)) {
  if (Button *listrow = listrow_find_mouse_over(data->region, event->xy)) {
    button_execute(C, data->region, listrow);
    return WM_UI_HANDLER_BREAK;
  }
}
```

这段解决的是“遮挡”问题。

场景是：

- 鼠标点到一颗 disabled 按钮
- 但这颗 disabled 按钮其实只是覆盖在某个 list row 上方
- 用户真正想点中的，往往是下面那整行

如果不做这个特判，disabled 按钮会挡住点击，导致底下的 list row 永远点不到。

所以这里的策略是：

1. 如果当前按钮是 disabled
2. 且用户做了左键按下
3. 就去当前位置再找一下有没有底下的 `listrow`
4. 如果有，就直接执行那一行

```cpp
button_execute(C, data->region, listrow);
return WM_UI_HANDLER_BREAK;
```

这里直接 `BREAK`，说明：

- 这次点击已经明确交给底下那一行处理了
- 不要再往当前 disabled 按钮的其他逻辑走

---

## disabled 状态下只允许 copy

```cpp
/* do copy first, because it is the only allowed operator when disabled */
if (do_copy) {
  if (but_copy(C, but, event->modifier & KM_ALT)) {
    return WM_UI_HANDLER_BREAK;
  }
}
```

这句注释点得很清楚：

- disabled 状态下，copy 是少数仍然允许的操作

原因也容易理解：

- copy 不会修改数据
- 它只是把当前值导出到剪贴板

所以即使按钮被禁用，复制它的内容通常仍然是安全合理的。

这里先处理 copy，有两个意义：

1. 它优先级高
2. 后面马上可能因为 `is_disabled` 提前返回，所以必须先处理

`but_copy(...)` 返回成功后，直接：

```cpp
return WM_UI_HANDLER_BREAK;
```

表示：

- 复制已经完成
- 这个事件被消费了

## 右键菜单处理

```cpp
if ((event->type == RIGHTMOUSE) && (event->modifier == 0) && (event->val == KM_PRESS)) {
```

这里只有最普通的右键按下才进来：

- `RIGHTMOUSE`
- 无修饰键
- `KM_PRESS`

### 先让某些 view item 成为“当前操作对象”

```cpp
auto *clicked_view_item_but = static_cast<ButtonViewItem *>(
    but->type == ButtonType::ViewItem ? but :
                                        view_item_find_mouse_over(data->region, event->xy));
if (clicked_view_item_but) {
  clicked_view_item_but->view_item->activate_for_context_menu(*C);
  ED_region_tag_redraw_no_rebuild(data->region);
}
```

这一段在做：

- 如果右键的是 `ViewItem`
- 或者当前位置能找到一个 `ViewItem`
- 那先把它设成 context menu 的当前目标项

这样做是为了确保：

- 后面弹出右键菜单时
- menu poll / menu operator 看到的 active item 是对的

否则容易出现：

- 菜单弹出来了
- 但菜单以为当前操作对象还是上一项

### 再尝试弹上下文菜单

```cpp
if (popup_context_menu_for_button(C, but, event)) {
  WM_cursor_modal_restore(data->window);
  return WM_UI_HANDLER_BREAK;
}
```

如果右键菜单成功打开：

- 恢复 modal cursor
- 直接 `BREAK`

因为此时右键事件已经完整消费掉了。

## 这段末尾：给 `disable_force` 的无边框文本按钮开一个编辑后门

```cpp
if (data->disable_force && but->type == ButtonType::Text &&
    ELEM(but->emboss, EmbossType::None, EmbossType::NoneOrStatus) &&
    event->type == LEFTMOUSE && (event->modifier & KM_CTRL || event->val == KM_DBL_CLICK))
{
  button_activate_state(C, but, BUTTON_STATE_TEXT_EDITING);
  return WM_UI_HANDLER_BREAK;
}
```

这是最近这次双击重命名支持里最关键的新分支之一。

### 先理解它为什么出现在这里

因为再往下马上就是：

```cpp
if (is_disabled) {
  return WM_UI_HANDLER_CONTINUE;
}
```

也就是说：

- 一旦当前按钮被视作 disabled
- 后面的正常文本按钮处理逻辑根本走不到

而这类 node socket rename label 恰好就是：

- hover 时允许激活
- 但又会被 `disable_force` 临时当 disabled

所以必须在这里、在 `is_disabled` 提前返回之前，给它留一个特例入口。

### 每个条件分别在卡什么

#### `data->disable_force`

只处理这类“逻辑上被强制 disabled”的按钮。

也就是说，这不是一般文本框通路，而是专门补那类 hover-only / label-like 文本按钮的洞。

#### `but->type == ButtonType::Text`

只对文本按钮生效。

因为目标是进入文本编辑，而不是让任意 disabled 按钮都吃掉双击。

#### `ELEM(but->emboss, EmbossType::None, EmbossType::NoneOrStatus)`

只对“无边框 / 近似 label 风格”的文本按钮生效。

这正是 node socket rename label 的关键特征之一。

#### `event->type == LEFTMOUSE`

只处理左键。

右键、键盘、滚轮等事件都不在这条特例里。

#### `(event->modifier & KM_CTRL || event->val == KM_DBL_CLICK)`

这里把两种允许进入编辑的手势并到了一起：

- `Ctrl+左键`
- `左键双击`

这样做的目的不是只新增双击，而是：

- 保住旧的 Ctrl+Click 语义
- 同时把双击纳入同一条“穿过 disable_force”的后门

### 进入编辑是怎么做的

```cpp
button_activate_state(C, but, BUTTON_STATE_TEXT_EDITING);
```

它不是简单打个标志，而是在切按钮状态机：

- 从 `BUTTON_STATE_HIGHLIGHT`
- 切到 `BUTTON_STATE_TEXT_EDITING`

进入这个状态后，后续才会建立文本编辑上下文，让输入光标和键盘编辑真正开始工作。

### 为什么这里直接 `return WM_UI_HANDLER_BREAK`

因为一旦决定进文本编辑：

- 这个左键事件已经有明确归属
- 不应该再继续往别的按钮/节点交互里传

所以必须立刻吃掉。

---

## 这整段代码的层次可以怎么记

从执行顺序看，这一段可以分成 6 层：

1. 取当前 active button 的运行时状态
2. 统一算出“当前是否按 disabled 语义处理”
3. 限定只在 `BUTTON_STATE_HIGHLIGHT` 下处理这一批逻辑
4. 先处理 copy/paste、list row 转发、disabled 点击透传
5. 再处理右键上下文菜单
6. 最后在 `is_disabled` 提前返回之前，为 `disable_force` 的文本按钮开特例入口

也就是说，这不是某个单一功能块，而是：

- **按钮在 hover/highlight 阶段收到事件后的统一前置分发层**

最近双击重命名加进去的那段代码，只是挂在这个分发层末尾的一个“高优先级特例”。

---

## 这段和双击修改接口名的关系

对 node socket rename 来说，这段最关键的是最后这块：

```cpp
if (data->disable_force && but->type == ButtonType::Text &&
    ELEM(but->emboss, EmbossType::None, EmbossType::NoneOrStatus) &&
    event->type == LEFTMOUSE && (event->modifier & KM_CTRL || event->val == KM_DBL_CLICK))
```

它解决的是：

- 这颗 socket name 按钮已经能被 hover 找到
- 但因为它平时不该像普通文本框一样吞掉点击，所以被 `disable_force`
- 这会让它在常规路径里过早退出

于是这里专门补了一条：

- `Ctrl+Click` 可以穿过去
- `Double-Click` 也可以穿过去

这就是为什么双击支持要改到 `do_button()` 这一层，而不只是改 `do_but_TEX(...)`。
